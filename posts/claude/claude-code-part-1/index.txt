2026년, 시장에는 수많은 AI가 쏟아졌다. 

단순 질의응답과 검색을 대신해주던 ChatGPT, 코드 스타일을 파악해 다음 줄을 예측해주는 GitHub Copilot에서 이제는 CLI에 직접 접속해 코드를 읽고, 수정하고, 실행까지 하는 Claude Code, Gemini CLI 같은 것들이 나왔다. 

개발 환경 자체에 AI가 들어온 것이다.

이 시대에 개발자 친구들에게도, 개발을 전혀 모르는 친구들에게도 심심찮게 듣는 말이 있다.

> 개발? AI로 딸깍하면 되는 거 아니야?

오늘은 매일 AI와 함께 개발을 하면서 느낀 나의 생각을 정리해본다.

---

## 웹마스터를 기억하는가

웹마스터는 1990년대 초, 웹이 막 태동하던 시절에 등장한 직업이다. 당시의 웹은 HTML과 링크로 이루어진 단순한 정적 페이지가 전부였다. HTML 몇 줄이면 웹사이트가 완성되는 구조였으니, 웹마스터 한 명이 디자인부터 코딩, 서버 관리까지 모두 혼자 감당할 수 있었다.

그러다 1990년대 말, 동적 웹이 등장하면서 흐름이 바뀌었다. 단순히 정보를 보여주는 데서 벗어나 사용자와 상호작용하는 웹으로 진화한 것이다. 2000년대에는 사용자가 직접 콘텐츠를 올리는 플랫폼 — Facebook, YouTube 같은 서비스들이 등장하면서 웹이 해야 할 일이 폭발적으로 늘었고, 그만큼 복잡도도 함께 증가했다. 복잡도가 높아질수록 각 분야의 깊은 지식과 경험을 가진 전문가들이 필요해졌고, 그 과정에서 웹마스터라는 직업은 자연스럽게 사라졌다.

하지만 웹마스터가 사라진 후 웹 관련 일자리는 오히려 늘었다. 분업화가 이루어지면서 프론트엔드 개발자, 백엔드 개발자, 데브옵스 엔지니어, UX 디자이너 등 훨씬 더 다양한 직군이 생겨났다.

AI가 등장한 지금, 비슷하면서도 다른 결과를 낳을 수 있는 질문이 떠오른다. *"이번엔 개발자 자체가 사라지는 것 아닐까?"*

나는 직업이 사라진다기보다는 '개발자'라는 단어의 의미가 재정립될 거라고 생각한다. 지금까지 개발자는 자신이 학습하고 경험한 지식을 코드로 풀어내는 사람, 프론트엔드면 프론트엔드, 백엔드면 백엔드 특정 분야의 전문가를 뜻했다. 그런데 AI는 이미 우리보다 방대한 지식과 경험을 갖추고 있다. 그렇다면 앞으로 개발자의 중심 역할은 달라질 수밖에 없다.

AI에게 시스템의 아키텍처와 코드의 방향성을 제시해주는 역할, 그것이 중심 역할이 되지 않을까?  실제로 GitHub의 CEO는 자신의 다음 직함이 **'코드 크리에이티브 디렉터'**가 될 거라고 말하기도 했다.

이렇게 생각하면 AI 시대에 잘해야 할 것도 명확해진다. 전반적인 플로우를 보는 능력, 적절한 추상화, 그리고 방향성 제시. 코드 한 줄보다 "이 시스템이 왜 이래야 하는지"를 설명할 수 있는 능력이 점점 더 중요해지고 있다.

## 무엇을 위임하고, 무엇을 쥐고 있어야 하는가

AI에게 넘길 수 있는 일과 넘기면 안 되는 일은 생각보다 명확하다.

**위임해도 되는 것들 — 기술과 실행의 영역:**

- **코딩과 보일러플레이트 작성.** 반복적인 코드 구조를 만드는 건 AI가 빠르고 정확하다. 컴포넌트의 기본 틀, API 호출 구조, CRUD 로직 같은 것들이 해당된다. 
- **테스트 케이스 생성.** 엣지 케이스를 빠짐없이 나열하는 건 사람보다 AI가 낫다. 다만 "어떤 시나리오를 테스트해야 하는가"를 정하는 건 사람의 몫이다.
- **단순 기능 구현과 버그 수정.** 명확한 스펙이 있는 작업, 에러 로그를 기반으로 한 디버깅은 AI에게 맡겨도 된다.
- **반복적인 작업.** 린트 수정, 마이그레이션 스크립트, 포맷 변환 등 규칙이 명확한 작업들이 해당된다. 

**위임하면 안 되는 것들 — 방향과 결정의 영역:**

- **아키텍처 결정.** 구조적 판단은 비즈니스 맥락, 팀 역량, 운영 비용 등 AI가 알기 어려운 요소들이 얽혀 있다.
- **최종 의사결정.** AI는 선택지를 제시해줄 수 있지만, 그중 무엇을 택할지는 결국 맥락을 아는 사람이 해야 한다.
- **정책 수립.** 에러 핸들링 정책, 보안 정책, 배포 전략 등은 조직의 상황과 우선순위에 따라 달라진다.
- **사용자 경험 판단.** "이 버튼이 여기 있는 게 맞나?"는 데이터와 직관, 사용자에 대한 이해가 필요한 영역이다. AI는 참고 의견을 줄 수 있을 뿐이다.

이 경계를 흐릿하게 두면 문제가 생긴다. AI가 짜준 아키텍처를 그대로 따라가다가, 나중에 요구사항이 바뀌었을 때 전체를 뜯어고쳐야 하는 상황이 올 수 있다. 위임은 하되, 판단은 내가 하는 것. 이 원칙이 중요하다.


## 요리조리 AI 활용해보기 

비개발자도 AI의 도움으로 개발할 수 있다고들 한다. 뭐 틀린 말은 아니다. 간단한 웹사이트나 프로토타입 수준에서는 충분히 가능하다. 

하지만 시스템이 복잡해지고, 다양한 예외 케이스를 고려해야 할수록 기본적인 개발 지식 없이는 진행이 어려워진다.

또한 기술 지식의 부족으로 잘못된 문제 해결 방법을 채택한다면, 다음 문제를 해결할 때 기존 코드 구조가 걸림돌이 되고 AI가 "일단 돌아가게" 만들어준 코드가 쌓이면, 결국 대대적인 수정이 필요해지고 비효율을 야기시킨다. 

그렇기 때문에 학습의 내용과 품질, 방향을 결정하고 지도하는 주체는 결국 사람, 특히 지식을 가지고 있는 전문가여야한다고 생각한다. 

고성능 AI는 애매한 질문이나 추상적인 요구사항에도 고품질의 답을 내놓는다. 하지만 그만큼 토큰을 많이 잡아먹는다, 즉 비용이 많이 든다. 반면 저렴한 모델은 비용은 싸지만 답변의 품질이 떨어진다.

그렇다면 성능이 낮은 AI는 쓸 수 없는 걸까? 그렇지 않다. 토큰 사용량에 맞춰 질문의 수준과 난이도를 낮추고 세부적으로 요구한다면, 저렴한 모델로도 충분히 원하는 품질의 답변을 얻을 수 있다. 

고성능 AI도 마찬가지로 가이드라인을 잘 짜주면 더 적은 토큰으로 더 효율적인 결과를 만들어낸다. 결국 어떤 모델을 쓰든, 가이드라인이 결과의 질을 결정한다.

## 작업 시 유의 사항

앞서 말했듯 가이드라인에 따라 결과물은 천차만별이다. 먼저 주의해야 할 것들:

**1. AI가 작업하는 동안 코드를 수정하지 말 것**

AI는 변경된 코드를 다시 분석해야 하기 때문에 시간과 토큰만 더 소모된다. 한 폴더, 즉 한 브랜치 안에서 여러 개의 AI를 동시에 작업시키는 것도 좋지 않다. 서로의 변경사항이 충돌하면 둘 다 헛수고가 된다.

**2. 작업을 잘게 쪼개서 분배할 것**

"로그인 기능 만들어줘"보다 "이메일 유효성 검사 함수 만들어줘"가 훨씬 정확한 결과를 낸다. 세분화할수록 AI가 이해하고 처리하기 쉽다. 이건 토큰 사용량을 줄이는 것이기도 하다.

**3. 질문은 구체적이고 명확하게**

무엇을 원하는지 모르면 나온 결과도 알 수 없다. "이거 좀 고쳐줘"보다 "이 함수에서 null 체크가 빠져있어서 TypeError가 난다. optional chaining으로 수정해줘"가 훨씬 좋은 결과를 만든다.

**4. AI의 학습을 돕는 문서화**

AI가 특정 작업에 대해 학습한 정도에 따라 생산성이 달라진다. 자주 사용하는 패턴이나 컴포넌트를 문서화해서 미리 학습시키면 매번 처음부터 설명할 필요가 없어진다. `CLAUDE.md`나 커스텀 스킬 파일이 좋은 예시다.

## 실전 팁

**1. 도메인별 문서화**

하나의 거대한 문서보다 도메인별로 잘게 쪼개는 게 낫다. AI가 필요한 정보를 더 빨리 찾을 수 있고, 불필요한 컨텍스트를 읽지 않아도 되니 토큰 사용량도 줄어든다.

**2. 문서 작성도 AI에게**

되도록 짧고 간결하게, 그리고 작성 과정 자체도 AI를 활용하자. 먼저 요구사항을 AI에게 학습시키고, 결과를 확인한 뒤 피드백해서 다시 정제하는 과정을 반복하면 된다. Plan 모드를 활용하면 이 과정이 더 체계적이 된다.

**3. 단순 작업에 적극 활용**

간단한 퍼블리싱이나 단순 작업은 AI가 사람보다 빠르고 정확할 수 있다. MCP 같은 도구를 잘 활용하면 반복 작업의 상당 부분을 자동화할 수 있다. 절약된 시간에 더 중요한 판단을 하자.

결국 흔히 말하는 "딸깍"은 없다. 해야할 것을 얼마나 정확히 알고 있는 가, 그리고 그것을 확실하게 요구하고 지시할 수 있느냐에 따라 결과물은 천차만별이기 때문이다. 같은 AI에게 같은 기능을 시켜도, 어떻게 지시했느냐에 따라 깔끔한 코드가 나올 수도 있고, 다른 요구사항이 들어오면 무너져내리는 모래성 같은 코드가 나올 수 도 있다.
