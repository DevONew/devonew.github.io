### 0. Context API란?

**React 내장 기능**으로 별도 라이브러리 없이 컴포넌트 트리 전체에 값을 전달할 수 있음. props drilling(부모→자식→자식으로 계속 내려주는 것) 없이 전역으로 값을 공유할 수 있음.

**Context API vs Zustand**

|                 | Context API          | Zustand            |
| --------------- | -------------------- | ------------------ |
| 설치            | React 내장           | 외부 라이브러리    |
| 용도            | 전역 값 전달         | 전역 상태 관리     |
| 리렌더링        | Consumer 전체 리렌더 | 구독한 값만 리렌더 |
| 비동기/devtools | ❌                   | ✅                 |

**Context API가 적합한 경우**

- 테마, 로케일, 유저 정보처럼 자주 안 바뀌는 전역 값
- 외부 라이브러리 추가하기 부담스러운 소규모 프로젝트
- 최근엔 컴파운드 패턴에서도 사용

**Zustand가 적합한 경우**

- 자주 바뀌는 상태 (장바구니, 필터, 모달 등)
- 리렌더 최적화가 중요한 경우
- 비동기 액션, devtools, persist 등이 필요한 경우

---

### 1. 왜 Context API를 썼는가

- 행사마다 `primaryColor`가 다르기 때문에 동적으로 테마를 적용해야 했음
- 여러 자식 컴포넌트(`ThemedButton` 등)에서 공통으로 색상을 참조해야 해서 Context API로 전역 상태 관리

---

### 2. 어떤 문제가 있었는가

**문제 1 — 초기값 깜빡임(Flash)**

- 행사 진입 페이지(`page.tsx`)에서 쿠키 검증 + 테마 색상 + 포스터 URL을 주는 API를 사용
- `primaryColor`를 `EventThemeProvider`에 주입하기 위해 `EventThemeInitializer` 컴포넌트가 별도로 필요했음
- `EventThemeInitializer`가 클라이언트 컴포넌트라 초기값(`#5435EB`)으로 먼저 렌더링 후 API 응답 받으면 색상이 바뀌는 깜빡임 발생

**문제 2 — page → layout으로 props 전달 불가**

- `primaryColor`는 `page.tsx`에서 API로 받아오는데, 이걸 부모인 `layout.tsx`로 올려보낼 방법이 없음
- Next.js 구조상 page에서 layout으로 props 전달이 불가능

**문제 3 — `useEventTheme()` 훅은 클라이언트 전용이지만 실제로는 문제 없었음**

- `ThemedButton`은 어차피 `'use client'` 클라이언트 컴포넌트라 `useEventTheme()` 훅 사용 자체는 문제 없었음
- 다만 Context API 구조 자체가 문제 1, 2를 해결할 수 없는 구조였음

---

### 3. 해결

**해결 1 — `primaryColor`만 반환하는 API 생성 + layout에서 주입**

- `primaryColor`만 반환하는 API(`/api/v1/participant/events/[eventId]/theme`) 생성
- 서버 컴포넌트인 `[eventId]/layout.tsx`에서 해당 API 조회 후 `primaryColor` 획득
- layout이 서버에서 렌더링될 때 이미 `primaryColor`가 주입된 HTML이 만들어져서 브라우저에 도달
- 클라이언트 마운트 후 색을 바꾸는 게 아니라 처음부터 올바른 색으로 그려지기 때문에 깜빡임 해결

**해결 2 — CSS 변수로 전환**

- Context API, Provider, Hook, Initializer 제거
- `<div style={{ '--primary-color': primaryColor }}>` 로 CSS 변수 주입
- 자식 컴포넌트는 `style={{ backgroundColor: 'var(--primary-color)' }}` 로 참조
- 클라이언트 상태 관리 없이 CSS 상속만으로 하위 컴포넌트 전체에 테마 색상 적용

**해결 2를 채택한 이유**

- 프로젝트에서 이미 Zustand를 사용 중인데 추가로 Context API까지 유지하는 건 오버엔지니어링이라고 판단
- 단순히 색상 하나를 하위 컴포넌트에 전달하기 위해 상태관리 툴을 쓸 필요가 없었음
- CSS 변수는 별도 라이브러리 없이 CSS 상속만으로 동일한 효과를 낼 수 있어서 채택

---
