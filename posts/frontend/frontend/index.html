아키텍처 설계론에 대해 공부해야겄어

  1. 컴포넌트 아키텍처

  2. 상태 관리 아키텍처 Flux/Redux: 단방향 데이터 흐름 (Action → Dispatcher → Store → View) Atomic State (Jotai, Recoil): 상태를 작은 atom 단위로 분리 Server State vs Client State: TanStack Query, SWR로 서버 상태 분리

  3. 폴더 구조 패턴 Feature-based: 기능별로 폴더 분리 (features/auth/, features/cart/) Layer-based: 역할별 분리 (components/, hooks/, services/) FSD (Feature-Sliced Design): 러시아에서 시작된 체계적 구조 (layers → slices → segments)

  4. 관심사 분리 Presentational vs Container: UI와 로직 분리 (현재는 hooks로 대체되는 추세) Custom Hooks: 재사용 가능한 로직 추출 Services Layer: API 호출, 비즈니스 로직을 별도 계층으로

  5. 의존성 관리 Dependency Injection: 테스트 용이성, 모듈 교체 유연성 Barrel Exports (index.ts): 모듈 진입점 단순화 Path Aliases: @/components처럼 깔끔한 import 경로

  6. 확장성 패턴 Plugin Architecture: 기능 확장을 위한 플러그인 시스템 Module Federation: 마이크로 프론트엔드, 독립적 배포 Monorepo: Turborepo, Nx로 패키지 관리 블로그 글 구성 제안 왜 아키텍처가 중요한가 - 유지보수성, 확장성, 팀 협업 컴포넌트 설계 원칙 - 단일 책임, 재사용성 상태 관리 전략 선택 - 프로젝트 규모별 접근 실제 폴더 구조 예시 - Feature-based 구조 추천 의존성 방향과 계층 - 상위 → 하위 단방향 의존
