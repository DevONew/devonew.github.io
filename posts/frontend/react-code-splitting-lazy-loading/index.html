React 코드 스플리팅과 Lazy Loading: 언제, 왜 써야 할까?

📅2026년 1월 12일
Reactfrontend성능최적화CodeSplitting

왜 코드 스플리팅이 필요한가?

React 앱을 배포하고 나면, 모든 JavaScript 코드가 하나의 큰 번들 파일로 합쳐집니다. 앱이 작을 때는 문제없지만, 기능이 추가될수록 번들 크기는 계속 커집니다.

문제는 사용자가 첫 화면을 보기 위해 앱 전체를 다운로드해야 한다는 것입니다.

로그인 페이지를 보려는 사용자가 관리자 대시보드, 설정 페이지, 통계 차트 라이브러리까지 모두 다운로드해야 한다면? 너무 낭비라 생각합니다.

이럴 때 필요한 코드만 필요한 시점에 로드하는 것이 코드 스플리팅!

코드 스플리팅이 필요한 신호

1. 번들 크기가 500KB를 넘어갈 때

# 빌드 후 번들 크기 확인
npm run build

# 출력 예시
File sizes after gzip:
  1.2 MB  build/static/js/main.abc123.js  #크다..
  50 KB   build/static/css/main.def456.css

일반적으로 gzip 압축 후 번들 크기가 500KB를 넘어가면, 코드 스플리팅을 고려해야 합니다. 모바일 3G 환경에서는 1MB 다운로드에 10초 이상 걸릴 수 있습니다.

2. 특정 페이지나 기능을 일부 사용자만 사용할 때

//  모든 사용자가 관리자 페이지 코드를 다운로드
import AdminDashboard from './pages/AdminDashboard';
import UserDashboard from './pages/UserDashboard';

function App() {
  return (
    <Routes>
      <Route path="/user" element={<UserDashboard />} />
      <Route path="/admin" element={<AdminDashboard />} />
    </Routes>
  );
}

일반 사용자는 관리자 페이지에 접근조차 하지 않는데, 해당 코드를 다운로드하는 건 낭비입니다.

3. 무거운 라이브러리를 사용할 때

// 차트가 필요 없는 페이지에서도 Chart.js 다운로드
import { Chart } from 'chart.js';  
import { PDFViewer } from 'react-pdf'; 
import { Editor } from '@monaco-editor/react';

차트, PDF 뷰어, 코드 에디터 같은 무거운 라이브러리는 필요한 페이지에서만 로드해야 합니다.

React.lazy()와 Suspense 사용법

기본 구조

import { lazy, Suspense } from 'react';

// 동적 import로 코드 스플리팅
const AdminDashboard = lazy(() => import('./pages/AdminDashboard'));

function App() {
  return (
    <Suspense fallback={<LoadingSpinner />}>
      <Routes>
        <Route path="/admin" element={<AdminDashboard />} />
      </Routes>
    </Suspense>
  );
}

lazy()로 감싼 컴포넌트는 별도의 청크 파일로 분리되어, 해당 라우트에 접근할 때만 다운로드됩니다.

라우트 기반 스플리팅

가장 흔하고 효과적인 패턴입니다.

import { lazy, Suspense } from 'react';
import { Routes, Route } from 'react-router-dom';

// 각 페이지를 별도로 분리
const Home = lazy(() => import('./pages/Home'));
const About = lazy(() => import('./pages/About'));
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Settings = lazy(() => import('./pages/Settings'));

function App() {
  return (
    <Suspense fallback={<PageLoader />}>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/about" element={<About />} />
        <Route path="/dashboard" element={<Dashboard />} />
        <Route path="/settings" element={<Settings />} />
      </Routes>
    </Suspense>
  );
}

빌드 결과:

main.js (50KB) - 앱 코어
home.js (30KB)
about.js (20KB)
dashboard.js (100KB)
settings.js (80KB)

이제 홈페이지 방문자는 50KB + 30KB = 80KB만 다운로드하면 됩니다. (기존: 280KB)

컴포넌트 레벨 스플리팅

특정 기능이 무거울 때 사용합니다.

import { lazy, Suspense, useState } from 'react';

// 차트 컴포넌트를 lazy로 분리
const HeavyChart = lazy(() => import('./components/HeavyChart'));

function Dashboard() {
  const [showChart, setShowChart] = useState(false);

  return (
    <div>
      <h1>대시보드</h1>

      <button onClick={() => setShowChart(true)}>
        차트 보기
      </button>

      {showChart && (
        <Suspense fallback={<div>차트 로딩 중...</div>}>
          <HeavyChart />
        </Suspense>
      )}
    </div>
  );
}

사용자가 "차트 보기" 버튼을 누를 때만 차트 라이브러리를 다운로드합니다.

언제 코드 스플리팅을 적용할까?

적용하면 좋은 경우

  1. 라우트별 분리

    • 모든 페이지를 한 번에 로드할 필요 없음
    • 가장 확실한 효과

  2. 권한별 분리

    const AdminPanel = lazy(() => import('./admin/AdminPanel'));
const UserPanel = lazy(() => import('./user/UserPanel'));

function Dashboard({ role }: { role: 'admin' | 'user' }) {
  return (
    <Suspense fallback={<Loading />}>
      {role === 'admin' ? <AdminPanel /> : <UserPanel />}
    </Suspense>
  );
}

  3. 무거운 라이브러리

    const PDFViewer = lazy(() => import('./components/PDFViewer'));
const VideoEditor = lazy(() => import('./components/VideoEditor'));
const CodeEditor = lazy(() => import('./components/CodeEditor'));

  4. 모달/다이얼로그

    const SettingsModal = lazy(() => import('./modals/SettingsModal'));

function App() {
  const [showSettings, setShowSettings] = useState(false);

  return (
    <>
      <button onClick={() => setShowSettings(true)}>설정</button>
      {showSettings && (
        <Suspense fallback={null}>
          <SettingsModal onClose={() => setShowSettings(false)} />
        </Suspense>
      )}
    </>
  );
}

적용하지 말아야 할 경우

  1. 작은 컴포넌트 (< 10KB)

    // 불필요한 분리 - 오히려 HTTP 요청만 늘어남
const Button = lazy(() => import('./components/Button'));

  2. 첫 화면에 바로 보이는 컴포넌트

    // 첫 화면 깜빡임 발생
const Header = lazy(() => import('./components/Header'));
const Hero = lazy(() => import('./components/Hero'));

  3. 자주 함께 사용되는 컴포넌트

    // 항상 같이 쓰인다면 분리할 필요 X
const Form = lazy(() => import('./components/Form'));
const FormInput = lazy(() => import('./components/FormInput'));
const FormButton = lazy(() => import('./components/FormButton'));

Suspense Fallback 전략

1. 페이지 레벨 로딩

function PageLoader() {
  return (
    <div className="flex items-center justify-center min-h-screen">
      <div className="animate-spin rounded-full h-12 w-12 border-b-2 border-blue-600" />
    </div>
  );
}

<Suspense fallback={<PageLoader />}>
  <Routes>
    <Route path="/" element={<Home />} />
  </Routes>
</Suspense>

2. 컴포넌트 레벨 로딩

function InlineLoader() {
  return (
    <div className="py-8 text-center">
      <div className="inline-block animate-pulse">로딩 중...</div>
    </div>
  );
}

<Suspense fallback={<InlineLoader />}>
  <HeavyComponent />
</Suspense>

3. 스켈레톤 UI

function ChartSkeleton() {
  return (
    <div className="space-y-4 animate-pulse">
      <div className="h-4 bg-gray-200 rounded w-3/4"></div>
      <div className="h-64 bg-gray-200 rounded"></div>
      <div className="h-4 bg-gray-200 rounded w-1/2"></div>
    </div>
  );
}

<Suspense fallback={<ChartSkeleton />}>
  <Chart data={data} />
</Suspense>

스켈레톤 UI는 실제 컴포넌트의 레이아웃과 비슷해서, 로딩 중에도 자연스러운 경험을 제공합니다.

프리로딩 전략

코드 스플리팅의 단점은 "로딩 시간"이 생긴다는 것입니다.

이를 해결하는 방법들은 다음과 같이 있습니다:

1. 마우스 호버 시 프리로드

import { lazy } from 'react';

const AdminPanel = lazy(() => import('./pages/AdminPanel'));

// 프리로드 함수
const preloadAdminPanel = () => {
  import('./pages/AdminPanel');
};

function Navigation() {
  return (
    <nav>
      <Link
        to="/admin"
        onMouseEnter={preloadAdminPanel}  // 호버 시 미리 로드
      >
        관리자 페이지
      </Link>
    </nav>
  );
}

링크에 마우스를 올리면 미리 다운로드하여, 클릭 시 즉시 표시됩니다.

2. 네트워크 유휴 시간 활용

import { useEffect } from 'react';

function App() {
  useEffect(() => {
    // 첫 화면 로드 후, 브라우저가 한가할 때 다음 페이지 
    if ('requestIdleCallback' in window) {
      requestIdleCallback(() => {
        import('./pages/Dashboard');
        import('./pages/Settings');
      });
    }
  }, []);

  return <Routes>...</Routes>;
}

3. 라우트 변경 예측

// 홈페이지에서 대부분 사용자가 대시보드로 이동한다면
function HomePage() {
  useEffect(() => {
    const timer = setTimeout(() => {
      import('./pages/Dashboard');
    }, 3000);

    return () => clearTimeout(timer);
  }, []);

  return <div>홈페이지</div>;
}

실전 예제: Before & After

Before: 번들 크기 1.2MB

import Chart from 'chart.js';
import { PDFViewer } from 'react-pdf';
import AdminDashboard from './pages/AdminDashboard';
import UserSettings from './pages/UserSettings';
import Reports from './pages/Reports';

function App() {
  return (
    <Routes>

문제점:

  • 첫 로드: 1.2MB (3G 환경 12초? 소요)
  • 모든 사용자가 관리자 페이지 코드 다운로드
  • Chart.js, PDF 라이브러리 항상 포함

After: 초기 번들 150KB

import { lazy, Suspense } from 'react';

const AdminDashboard = lazy(() => import('./pages/AdminDashboard'));
const UserSettings = lazy(() => import('./pages/UserSettings'));
const Reports = lazy(() => import('./pages/Reports'));

function App() {
  return (
    <Suspense fallback={<PageLoader />}>
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/admin" element={<AdminDashboard />} />
        <Route path="/settings" element={<UserSettings />} />
        <Route path="/reports" element={<Reports />} />
      </Routes>
    </Suspense>
  );
}

개선 결과:

  • 첫 로드: 150KB (3G 환경 1.5로 단축)
  • 관리자만 AdminDashboard 코드 다운로드
  • 차트와 PDF는 Reports 페이지에서만 로드

성능 측정하기

코드 스플리팅 효과를 확인하는 방법:

1. Webpack Bundle Analyzer

npm install --save-dev webpack-bundle-analyzer

// next.config.js (Next.js의 경우)
const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
});

module.exports = withBundleAnalyzer({
  // ...config
});

ANALYZE=true npm run build

번들 크기를 시각적으로 확인할 수 있습니다.

2. Lighthouse 성능 측정

Chrome DevTools > Lighthouse 탭에서:

  • First Contentful Paint (FCP)
  • Largest Contentful Paint (LCP)
  • Time to Interactive (TTI)

코드 스플리팅 전후로 이 지표들이 개선되는지 확인합니다.

3. Network 탭 확인

Chrome DevTools > Network 탭에서:

  • 초기 로드 시 다운로드되는 파일 개수와 크기
  • 페이지 이동 시 추가 다운로드되는 청크 파일들

마치며

적용 기준:

  • 번들 크기가 500KB 이상일 때
  • 특정 기능이 일부 사용자에게만 필요할 때
  • 무거운 라이브러리를 사용할 때

주의사항:

  • 너무 작은 컴포넌트는 분리하지 말 것
  • 첫 화면에 필요한 컴포넌트는 분리하지 말 것
  • 로딩 상태(Suspense fallback)를 항상 제공할 것

Next.js나 Vite 같은 최신 도구들은 대부분 자동으로 코드 스플리팅을 해주지만, 라이브러리 레벨이나 컴포넌트 레벨에서는 직접 판단해야 합니다.
