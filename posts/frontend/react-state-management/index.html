모던 React 상태 관리에 대하여

📅2026년 1월 7일
React상태관리ZustandJotaiRecoilReduxContext API

들어가며

React 애플리케이션을 개발하다 보면 반드시 마주치게 되는 문제가 있습니다. 바로 상태 관리입니다.

처음에는 간단해 보입니다. React가 제공하는 useState로 충분하죠:

function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}

하지만 애플리케이션이 성장하면서 문제가 시작됩니다. 로그인한 사용자 정보를 Header, Sidebar, Profile 페이지에서 모두 사용해야 한다면? 장바구니 상태를 상품 목록과 결제 페이지에서 동시에 관리해야 한다면?

function App() {
  const [user, setUser] = useState(null);

  return <Layout user={user} setUser={setUser}>
    <Sidebar user={user}>
      <UserMenu user={user}>
        <UserProfile user={user} onLogout={() => setUser(null)} />
      </UserMenu>
    </Sidebar>
  </Layout>;
}

useState는 로컬 상태를 관리하는 데는 훌륭하지만, 컴포넌트 트리 전체에서 공유해야 하는 전역 상태를 다루기에는 적합하지 않습니다. 상태를 최상위 컴포넌트로 끌어올리고, props로 전달하고, 중간 컴포넌트들은 사용하지도 않는 데이터를 전달만 하는 역할이 되어버립니다.

이것이 바로 상태 관리 라이브러리가 필요한 이유입니다.

2016년 Redux가 등장한 이후, React 생태계의 상태 관리는 빠르게 진화해왔습니다. 2019년 Hooks의 등장으로 Context API가 실용적으로 사용 가능해졌고, 이후 Zustand, Jotai, Recoil 같은 현대적인 라이브러리들이 등장하면서 개발자들에게 더 많은 선택지가 생겼습니다.

그렇다면 2026년 현재, 우리는 어떤 상태 관리 솔루션을 선택해야 할까요?

왜 상태 관리가 필요한가?

React의 기본 원칙은 단방향 데이터 흐름입니다. 부모 컴포넌트에서 자식 컴포넌트로 props를 전달하는 구조죠. 하지만 애플리케이션이 커지면서 다음과 같은 문제에 직면하게 됩니다:

1. Props Drilling 문제

<App user={user}>
  <Layout user={user}>
    <Sidebar user={user}>
      <UserMenu user={user}>
        <UserProfile user={user} />
      </UserMenu>
    </Sidebar>
  </Layout>
</App>

중간 컴포넌트들은 user 데이터를 사용하지도 않으면서 단지 전달만 하고 있습니다. 코드의 유지보수성이 떨어지고, 컴포넌트의 재사용성도 저하됩니다.

2. 컴포넌트 간 상태 공유

로그인한 사용자 정보, 테마 설정, 쇼핑몰의 장바구니처럼 여러 컴포넌트에서 동시에 접근하고 수정해야 하는 데이터가 있습니다. 이런 데이터를 각 컴포넌트의 로컬 상태로 관리하면 동기화 문제가 발생합니다.

3. 복잡한 상태 로직

// 복잡해지는 useState와 useEffect
const [data, setData] = useState(null);
const [loading, setLoading] = useState(false);
const [error, setError] = useState(null);
const [filters, setFilters] = useState({});
const [sorting, setSorting] = useState('asc');

useEffect(() => {
  // 데이터 페칭 로직
}, [filters, sorting]);

useEffect(() => {
  // 또 다른 side effect
}, [data]);

비즈니스 로직이 복잡해질수록 useState와 useEffect가 난무하게 되고, 상태 업데이트의 흐름을 추적하기 어려워집니다.

4. 성능 최적화

불필요한 리렌더링을 방지하기 위해 React.memo, useMemo, useCallback을 도입하지만, 이것만으로는 한계가 있습니다. 특히 전역 상태가 변경될 때 해당 상태를 사용하지 않는 컴포넌트까지 리렌더링되는 문제를 해결하기는 쉽지 않습니다.

상태 관리의 두 가지 관점

상태 관리를 논할 때 반드시 구분해야 할 개념이 있습니다:

  • 클라이언트 상태 (Client State): UI 상태, 폼 입력값, 모달 열림/닫힘 등 클라이언트에서만 관리하는 상태
  • 서버 상태 (Server State): API로부터 받아온 데이터, 캐싱이 필요한 원격 데이터

과거에는 Redux로 모든 것을 관리했지만, 지금은 서버 상태는 TanStack Query,React Query나 SWR로, 클라이언트 상태는 가벼운 라이브러리로 분리하는 것이 일반적입니다.

이제 본격적으로 각 상태 관리 라이브러리의 특징과 사용 사례를 살펴보겠습니다.

1. Context API - React의 내장 솔루션

Context API는 별도의 라이브러리 없이 전역 상태를 관리할 수 있는 방법입니다.

특징

  • 추가 의존성 없음 (React 내장)
  • 간단한 전역 상태 관리에 적합
  • Provider/Consumer 패턴

코드 예제

// 1. Context 생성
const UserContext = createContext(null);

// 2. Provider로 감싸기
function App() {
  const [user, setUser] = useState(null);

  return (
    <UserContext.Provider value={{ user, setUser }}>
      <Header />
      <MainContent />
    </UserContext.Provider>
  );
}

// 3. 어디서든 useContext로 접근
function Header() {
  const { user, setUser } = useContext(UserContext);

  return (
    <header>
      {user ? (
        <button onClick={() => setUser(null)}>로그아웃</button>
      ) : (
        <button onClick={() => setUser({ name: '사용자' })}>로그인</button>
      )}
    </header>
  );
}

한계점

const ThemeContext = createContext();

function App() {
  const [theme, setTheme] = useState('light');
  const [user, setUser] = useState(null);

  // theme이나 user 중 하나만 바뀌어도
  // 모든 Consumer가 리렌더링됩니다
  return (
    <ThemeContext.Provider value={{ theme, setTheme, user, setUser }}>
      <ComponentA /> {/* theme만 사용 */}
      <ComponentB /> {/* user만 사용 */}
    </ThemeContext.Provider>
  );
}

theme이 변경되면 user만 사용하는 컴포넌트도 리렌더링됩니다. 이를 해결하려면 Context를 분리하거나 useMemo를 과도하게 사용해야 하는데, 이는 코드를 복잡하게 만듭니다.

언제 사용할까?

  • 테마, 언어 설정 같은 단순한 전역 상태
  • 자주 변경되지 않는 데이터
  • 작은 규모의 앱

2. Zustand

Zustand는 독일어로 "상태"를 의미하며, 이름처럼 상태 관리에 집중한 가볍고 직관적인 라이브러리입니다.

특징

  • 보일러플레이트가 거의 없음
  • 번들 크기가 작음 (1KB)
  • Provider 없이 사용 가능
  • 미들웨어 지원 (persist, devtools 등)

코드 예제

import { create } from 'zustand';

// 1. 스토어 생성 
const useUserStore = create((set) => ({
  user: null,
  setUser: (user) => set({ user }),
  logout: () => set({ user: null }),
}));

// 2. 어디서든 훅처럼 사용
function Header() {
  const user = useUserStore((state) => state.user);
  const logout = useUserStore((state) => state.logout);

  return (
    <header>
      {user ? (
        <button onClick={logout}>로그아웃</button>
      ) : (
        <button>로그인</button>
      )}
    </header>
  );
}

// 3. 필요한 상태만 구독 
function UserProfile() {
  // user가 변경될 때만 리렌더링
  const user = useUserStore((state) => state.user);
  return <div>{user?.name}</div>;
}

예제: 장바구니

const useCartStore = create((set, get) => ({
  items: [],

  addItem: (product) => set((state) => ({
    items: [...state.items, { ...product, quantity: 1 }]
  })),

  removeItem: (productId) => set((state) => ({
    items: state.items.filter(item => item.id !== productId)
  })),

  updateQuantity: (productId, quantity) => set((state) => ({
    items: state.items.map(item =>
      item.id === productId ? { ...item, quantity } : item
    )
  })),

  // getter 함수도 정의 가능
  getTotalPrice: () => {
    const items = get().items;
    return items.reduce((sum, item) => sum + item.price * item.quantity, 0);
  },

  clearCart: () => set({ items: [] }),
}));

// 사용
function Cart() {
  const items = useCartStore((state) => state.items);
  const removeItem = useCartStore((state) => state.removeItem);
  const getTotalPrice = useCartStore((state) => state.getTotalPrice);

  return (
    <div>
      {items.map(item => (
        <div key={item.id}>
          {item.name} x {item.quantity}
          <button onClick={() => removeItem(item.id)}>삭제</button>
        </div>
      ))}
      <div>총 금액: {getTotalPrice()}원</div>
    </div>
  );
}

Persist 미들웨어

localStorage나 sessionStorage에 상태를 자동으로 저장하고 복원해주는 미들웨어입니다. 페이지를 새로고침해도 상태가 유지됩니다.

import { create } from 'zustand';
import { persist } from 'zustand/middleware';

// localStorage에 자동 저장/복원
const useSettingsStore = create(
  persist(
    (set) => ({
      theme: 'light',
      language: 'ko',
      setTheme: (theme) => set({ theme }),
      setLanguage: (language) => set({ language }),
    }),
    {
      name: 'app-settings',
    }
  )
);

언제 사용할까?

  • 대부분의 경우
  • 간단하고 빠른 전역 상태 관리가 필요할 때
  • Redux는 과하고 Context는 부족할 때
  • 번들 크기를 최소화하고 싶을 때

3. Recoil

Recoil은 Facebook에서 만든 상태 관리 라이브러리로, React의 내부 동작 방식을 깊이 이해하고 설계되었습니다.

특징

  • Atom과 Selector를 사용한 세분화된 상태 관리
  • React Concurrent Mode와 완벽한 호환
  • Devtools 지원
  • 비동기 상태 처리에 강점

코드 예제

import { atom, selector, useRecoilState, useRecoilValue } from 'recoil';

// 1. Atom 정의
const userState = atom({
  key: 'userState', 
  default: null,
});

const todoListState = atom({
  key: 'todoListState',
  default: [],
});

// 2. Selector (derived state)
const completedTodosSelector = selector({
  key: 'completedTodosSelector',
  get: ({ get }) => {
    const todos = get(todoListState);
    return todos.filter(todo => todo.completed);
  },
});

// 3. 사용
function TodoStats() {
  const completedTodos = useRecoilValue(completedTodosSelector);

  return <div>완료된 할 일: {completedTodos.length}개</div>;
}

function UserProfile() {
  const [user, setUser] = useRecoilState(userState);

  return (
    <div>
      {user ? user.name : '로그인 필요'}
      <button onClick={() => setUser(null)}>로그아웃</button>
    </div>
  );
}

비동기 Selector

const currentUserQuery = selector({
  key: 'currentUserQuery',
  get: async ({ get }) => {
    const userId = get(userIdState);
    const response = await fetch(`/api/users/${userId}`);
    return response.json();
  },
});

// Suspense와 함께 사용
function CurrentUserInfo() {
  const user = useRecoilValue(currentUserQuery);
  return <div>{user.name}</div>;
}

function App() {
  return (
    <RecoilRoot>
      <Suspense fallback={<div>로딩중...</div>}>
        <CurrentUserInfo />
      </Suspense>
    </RecoilRoot>
  );
}

언제 사용할까?

  • Facebook/Meta 생태계와의 통합이 필요할 때
  • React Concurrent Mode를 적극 활용하는 앱
  • 복잡한 비동기 의존성이 있는 경우
  • 실험적 기능을 사용해도 괜찮을 때 (아직 1.0 미만)

4. Jotai

Jotai는 Recoil에서 영감을 받아 만들어진 라이브러리로, 상태를 atom이라는 작은 단위로 쪼개서 관리합니다. Recoil보다 더 가볍고 간단합니다.

특징

  • Bottom-up 접근 (atom 조합)
  • 불필요한 리렌더링 최소화
  • TypeScript와 궁합이 좋음
  • React Suspense와 자연스러운 통합

코드 예제

import { atom, useAtom } from 'jotai';

// 1. atom 정의 (primitive values)
const userAtom = atom(null);
const themeAtom = atom('light');

// 2. derived atom (computed value)
const userNameAtom = atom(
  (get) => get(userAtom)?.name ?? 'Guest'
);

// 3. 사용
function Header() {
  const [user, setUser] = useAtom(userAtom);
  const [userName] = useAtom(userNameAtom); // read-only

  return (
    <header>
      안녕하세요, {userName}님
      <button onClick={() => setUser(null)}>로그아웃</button>
    </header>
  );
}

function ThemeToggle() {
  const [theme, setTheme] = useAtom(themeAtom);

  // user 상태 변경과 무관하게 독립적으로 동작
  return (
    <button onClick={() => setTheme(theme === 'light' ? 'dark' : 'light')}>
      테마 변경
    </button>
  );
}

Async Atom

import { atom } from 'jotai';

// 비동기 atom
const userIdAtom = atom(1);

const userAtom = atom(async (get) => {
  const userId = get(userIdAtom);
  const response = await fetch(`/api/users/${userId}`);
  return response.json();
});

// Suspense와 함께 사용
function UserProfile() {
  const [user] = useAtom(userAtom); // Suspense가 로딩 처리
  return <div>{user.name}</div>;
}

function App() {
  return (
    <Suspense fallback={<div>로딩중...</div>}>
      <UserProfile />
    </Suspense>
  );
}

Zustand vs Jotai

// Zustand: Top-down (하나의 큰 store)
const useStore = create((set) => ({
  user: null,
  theme: 'light',
  cart: [],
  // 모든 상태가 한 곳에
}));

// Jotai: Bottom-up (작은 atom들의 조합)
const userAtom = atom(null);
const themeAtom = atom('light');
const cartAtom = atom([]);
// 필요한 atom만 import해서 사용

언제 사용할까?

  • 상태가 독립적으로 존재하고 조합되는 경우
  • React Suspense를 적극 활용하고 싶을 때
  • 세밀한 리렌더링 제어가 필요할 때
  • TypeScript를 사용하는 프로젝트

5. TanStack Query (React Query) - 서버 상태 관리의 표준

TanStack Query는 클라이언트 상태가 아닌 서버 상태를 관리하는 라이브러리입니다.

특징

  • 자동 캐싱, 백그라운드 업데이트
  • Optimistic updates
  • 무한 스크롤, 페이지네이션 지원
  • 전역 상태 라이브러리와 함께 사용

코드 예제

import { useQuery, useMutation, useQueryClient } from '@tanstack/react-query';

// 1. 데이터 페칭
function UserProfile({ userId }) {
  const { data: user, isLoading, error } = useQuery({
    queryKey: ['user', userId],
    queryFn: () => fetch(`/api/users/${userId}`).then(res => res.json()),
    staleTime: 5000, // 5초간 fresh
  });

  if (isLoading) return <div>로딩중...</div>;
  if (error) return <div>에러 발생!</div>;

  return <div>{user.name}</div>;
}

// 2. Mutation
function UpdateUserButton({ userId }) {
  const queryClient = useQueryClient();

  const mutation = useMutation({
    mutationFn: (newName) =>
      fetch(`/api/users/${userId}`, {
        method: 'PATCH',
        body: JSON.stringify({ name: newName }),
      }),
    onSuccess: () => {
      // 캐시 무효화 및 리페칭
      queryClient.invalidateQueries(['user', userId]);
    },
  });

  return (
    <button onClick={() => mutation.mutate('새 이름')}>
      이름 변경
    </button>
  );
}

// 3. Optimistic Updates
const mutation = useMutation({
  mutationFn: updateTodo,
  onMutate: async (newTodo) => {
    // 이전 쿼리 취소
    await queryClient.cancelQueries(['todos']);

    // 이전 값 백업
    const previousTodos = queryClient.getQueryData(['todos']);

    // Optimistic update
    queryClient.setQueryData(['todos'], (old) => [...old, newTodo]);

    return { previousTodos };
  },
  onError: (err, newTodo, context) => {
    // 실패 시 롤백
    queryClient.setQueryData(['todos'], context.previousTodos);
  },
});

언제 사용할까?

  • API 데이터를 다루는 모든 경우
  • 캐싱, 동기화, 백그라운드 업데이트가 필요할 때
  • Redux로 서버 상태를 관리하고 있다면 꼭 고려해보세요
